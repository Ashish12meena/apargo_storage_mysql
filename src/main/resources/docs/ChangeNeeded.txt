# Storage Service â€” Batch Media Upload Design

## Problem Statement

During Facebook template sync, each media file triggers an **individual HTTP POST** to the storage service. For a sync with 12 files, this causes:

- **Optimistic lock contention** on `project_storage` and `org_storage` rows
- Repeated `ObjectOptimisticLockingFailureException` with exponential backoff retries
- **12 separate transactions**, 12 quota reservations, 12 media inserts
- Total sync time: **~25 seconds** for what should take 3-5 seconds

---

## 1. New Endpoint: `POST /api/v1/media/upload/batch`

**Location:** `MediaController.java`

| Aspect | Detail |
|---|---|
| Method | `POST` |
| Path | `/api/v1/media/upload/batch` |
| Content-Type | `multipart/form-data` |
| Request | List of `MultipartFile` (param name: `files`) |
| Headers | `X-Org-Id`, `X-Project-Id`, `X-Waba-Id` |
| Response | `ApiResponse<BatchMediaUploadResponse>` |

**Request contract:**
- `files` â€” multiple file parts (same param name repeated)
- All files must belong to the **same org + project** (enforced by headers)

**Response contract:**
- Returns a list of results in the **same order** as the input files
- If an individual file fails validation, that entry has an error while others succeed
- If quota is insufficient for the total batch size, the entire request is rejected upfront

## 2. New Service: `BatchMediaUploadService`

**Location:** `service/impl/media/BatchMediaUploadService.java`

This is the core orchestrator for batch uploads. The flow:

```
Receive List<MultipartFile>

        â–Ľ

        Reject invalid files, collect valid ones
        â”‚
        â–Ľ

        â”‚
        â–Ľ

        Single atomic UPDATE on project_storage
        Single atomic UPDATE on org_storage
        â”‚
        â–Ľ
Step 4: Convert all MultipartFiles to temp files
        â”‚
        â–Ľ

        Use CompletableFuture or internal thread pool
        All parallelism is WITHIN the service â€” no external contention
        â”‚
        â–Ľ
Step 6: Build list of Media entities
        â”‚
        â–Ľ
Step 7: Batch insert all Media entities
        mediaRepository.saveAll(mediaList)
        â”‚
        â–Ľ
Step 8: Cleanup all temp files
        â”‚
        â–Ľ

```

**Key design points:**

- Quota is reserved **once** for the total â€” not per file
- File I/O (saving to local/S3) runs in parallel internally
- Database insert is a **single `saveAll()`** call
- If file storage fails for some files mid-batch:
  - Successfully stored files are kept
  - Failed files are reported in the response
  - Quota is adjusted: release the bytes for failed files only

## 3. Atomic Quota Reservation (Replaces Optimistic Locking)

**Location:** `OptimisticQuotaService.java` and repository layer

Replace the read-modify-write pattern with **atomic SQL UPDATEs**.

**New repository methods:**

| Repository | Method | Purpose |
|---|---|---|
| `ProjectStorageRepository` | `incrementUsage(orgId, projectId, totalSize)` | Atomic `UPDATE SET used_bytes = used_bytes + ?` with capacity check in WHERE clause |
| `ProjectStorageRepository` | `decrementUsage(orgId, projectId, totalSize)` | Atomic `UPDATE SET used_bytes = GREATEST(0, used_bytes - ?)` |
| `OrgStorageRepository` | `incrementUsage(orgId, totalSize)` | Same pattern at org level |
| `OrgStorageRepository` | `decrementUsage(orgId, totalSize)` | Same pattern at org level |

**How it works:**
- The WHERE clause includes `AND used_bytes + :size <= max_bytes`
- If the update affects 0 rows â†’ quota exceeded (or row not found)
- No `@Version` needed for this path â€” the UPDATE is inherently atomic
- No `@Retryable` needed â€” no optimistic lock failures possible
- Runs in `REQUIRES_NEW` transaction so it commits independently

**What to keep:**
- Keep `@Version` on entities for the pessimistic `QuotaService` (used by delete/release flows)
- Keep the nightly reconciliation job as a safety net

## 4. Batch Configuration

**Location:** `application.yml` + `MediaProperties.java`

| Property | Default | Purpose |
|---|---|---|
| `media.service.batch.max-files` | `20` | Safety cap â€” max files per batch request |

| Property | New Value |

## 5. Validation Changes

**Location:** `MediaValidator.java`

Add a batch validation method:

- Validates each file individually (MIME type, size, filename)
- Validates aggregate constraints (total count, total size)
- Returns a result object with valid files + list of rejected files with reasons
- Does NOT throw on individual file failures â€” collects them

## 6. Response Structure for Batch

**New DTOs:**

```
BatchMediaUploadResponse:
  - successCount: int
  - failedCount: int
  - results: List<BatchFileResult>

BatchFileResult:
  - originalFilename: String
  - status: SUCCESS | FAILED
  - url: String (null if failed)
  - mediaType: MediaType (null if failed)
  - contentType: String
  - fileSizeBytes: Long
  - error: String (null if success)
```

---

## 7. Error Handling

**Upload phase:**
- If batch succeeds but individual files within the batch fail â†’ only successful files are persisted, failed ones reported in response
- If quota is insufficient for the total batch â†’ entire batch rejected with 400
- If file storage fails mid-batch â†’ successfully stored files kept, quota released for failed bytes only

**Rollback behavior:**
- Storage service handles its own quota rollback internally
- No external coordination needed

---

## 8. Edge Cases

| Scenario | Handling |
|---|---|
| Batch exceeds max files limit (>20) | Reject with 400 |
| Batch exceeds max total size | Reject entire batch with 400 |
| Quota sufficient for some files but not all | Reject entire batch â€” atomic all-or-nothing |
| Individual file fails validation | Exclude from batch, report in response |
| File storage fails mid-batch | Track which succeeded, release quota for failed bytes |
| Network timeout on batch upload | Storage service rolls back quota |
